# POS-Backup Project Guide (for AI briefing — no code access)

Generated: 2026-07-08. This file is a structural map of a Laravel 11 multi-tenant
POS / ERP / HRM / Contracting SaaS application, intended to brief a fresh AI
conversation so it can help plan improvements without direct code access.

================================================================================
## 1. OVERVIEW
================================================================================

This is a multi-tenant Point-of-Sale + ERP + HRM + Contracting management SaaS
built on Laravel 11 with the `stancl/tenancy` package (single-database-per-tenant
style multi-tenancy, domain-identified). The app has three main "sides":

1. **Central website** (marketing site + tenant registration + pricing/checkout)
   — routes/central.php, app/Http/Controllers/Central/Site
2. **CPanel** (super-admin backoffice for the SaaS owner: manage tenants, plans,
   subscriptions, blogs, pages, partners, translations, payment methods)
   — routes/central_panel.php, app/Http/Controllers/Central/CPanel,
     app/Livewire/Central/CPanel
3. **Tenant app** (the actual POS/ERP product used by each tenant business):
   POS sales, purchases, stock, HRM (attendance/payroll/leaves/claims),
   Contracting/construction module (projects, BOQs, tenders, purchase orders,
   equipment, timesheets, chart of accounts, journal entries), fixed assets,
   reports (financial/sales/purchases/inventory/tax/performance), and an
   Employee self-service portal.
   — routes/tenant.php, app/Http/Controllers/Tenant, app/Http/Controllers/Admin,
     app/Livewire/Admin/*, app/Livewire/Employee/*, app/Models/Tenant/*

The tenant app appears to have **at least 3 parallel view-layer implementations**
of the same UI: `resources/views/livewire/admin`, `resources/views/livewire/hud`,
and `resources/views/livewire/tenant-tailwind-gemini` — each mirroring the same
folder structure (accounts, branches, contracting, hrm, reports, etc.) with the
same Livewire PHP classes presumably switching Blade view based on a
`defaultLayout`/theme setting (see AppServiceProvider `View::share('defaultLayout', ...)`).
This tripling of view code is worth flagging for consolidation.

Business domains covered: multi-branch retail/POS, purchasing, stock/inventory
(with stock transfers & stock-taking/adjustments), sales, refunds, sale/purchase
"requests" (quotation-like workflow before conversion), discounts, taxes (VAT),
accounts & transactions (double-entry-ish ledger), checks (cheque) management,
fixed assets + depreciation, expenses & expense categories, cash registers,
HRM (employees, departments, designations, contracts, attendance, leave,
expense claims, payroll runs/slips), and a full "Contracting" ERP module
(tenders, BOQs, supplier quotations, projects, contracts, extracts,
purchase requests/orders, inventory, warehouses, items, chart of accounts,
cost centers, journal entries, workers, timesheets, equipment/equipment logs).

Central side also has: blogs (heavily used — dozens of migrations seed blog
content, apparently for SEO), pages, FAQs, sliders, partners & partner
commissions, plans & subscriptions (with payment methods integration:
Paymob, PayPal), register requests (tenant sign-up approval flow), countries/
currencies/languages (i18n + currency conversion), contact form, SEO
(ralphjsmit/laravel-seo), sitemap generation.

## 3. MULTI-TENANCY (stancl/tenancy)
================================================================================

Package: `stancl/tenancy` ^3.8. Two config files:
- `config/tenancy.php` — main tenancy config. `tenant_model` => `App\Models\Tenant`.
  Has `central_domains` array (domains NOT treated as tenants — central website).
  Has a `database` connection/tenant-DB config block.
- `config/tenancy_live.php` — looks like a second/production-specific tenancy
  config variant (worth checking why two files exist / whether one is dead code).

Identification: **domain-based** — `App\Http\Middleware\InitializeTenancyByDomain`
extends stancl's `IdentificationMiddleware`, applied first in the
`routes/tenant.php` middleware group alongside `PreventAccessFromCentralDomains`.

`App\Models\Tenant` extends stancl's `BaseTenant` and implements
`TenantWithDatabase` — i.e. **one database per tenant**.

`app/Providers/TenancyServiceProvider.php` wires: tenant created/deleted events
(likely provisioning/tearing down the tenant DB), tenancy bootstrapped/ended
hooks, custom Livewire update-route handling per tenant, a `View::composer('*')`
global view composer, and forces tenancy middleware to highest priority
(`makeTenancyMiddlewareHighestPriority`).

Tenant-specific migrations live in `database/migrations/tenant/` (~70 files,
run against each tenant's own database) vs. central migrations directly under
`database/migrations/` (tenants/domains tables, countries/currencies/languages,
plans/subscriptions, blogs, cpanel admins, payment methods, etc.)

Console commands with tenant awareness: `ClearTenantCache`, `TenantCreateAdmin`,
`Console/Commands/Tenant/CleanAllSoftDeletes`, `StockQuantityAlertCheck`,
`SalesSummaryReport`, `CustomersDueAmountsAlert`, `SupplierDueAmountsAlert` (these
iterate tenants and run scheduled per-tenant jobs/alerts/emails).

## 5. ROUTES SUMMARY
================================================================================

### routes/web.php (11 lines)
Not a real route file — just iterates `config('tenancy.central_domains')` and
`Route::domain($domain)->group()` to include `routes/central.php` for each
configured central domain. All actual central routes live in central.php.

### routes/central.php (54 lines) — public marketing site
Single top-level group: `prefix '/'`, middleware `[SiteTranslationMiddleware,
RedirectToSecure]`. Routes (unnamed group, all under central domains):
- `register` (GET/POST, throttled 5/10) -> RegisterController@register/postRegister
- `register/accept/{id}` -> acceptRegistration
- `contact`, `contact-us` -> HomeController
- `checkout/{token?}` -> Livewire CheckoutPage
- `pricing`, `pricing/compare` -> PricingPage / HomeController@pricingCompare
- `faqs`, `blogs`, `blogs/{slug}` (no-lang fallback versions)
- Nested lang-prefixed group `{lang?}` (regex-constrained to en|ar[-xx]) with
  `as => lang.`, middleware `CentralWebsitesetLocal`: localized blogs, faqs,
  `{slug}` static page, and `/` -> `central-home`
- `lang/{locale}` -> changeLanguage
- `payment/{type}` group (type constrained check|success|failed) -> PaymentController callback/paymentCallbackPage
- catch-all `{slug}` -> PageController@renderPage (regex excludes cpanel/faqs/blogs/pricing/register/payment)
- `require __DIR__.'/central_panel.php'` at the end

### routes/central_panel.php (77 lines) — CPanel backoffice
Single group: `prefix 'cpanel', as 'cpanel.', middleware [CpanelTranslationMiddleware]`.
- Public: `login` (GET/POST), `logout`
- Nested group `middleware ['auth:' . CPANEL_ADMINS_GUARD]` containing ~35 GET
  routes, mostly `ComponentClass::class` (full-page Livewire routes) rather than
  controller actions: dashboard, admins.list, blogs (list/create/edit), pages
  (list/create/edit), faqs (list/create/edit), file-manager, contacts.list,
  countries/currencies/languages.list, translations, subscriptions.list,
  plans.list, features.list, register-requests.list, customers (list/create/details),
  sliders.list, partners (list/create/edit), partner-commissions.list,
  payment-methods (list/create/edit).

### routes/tenant.php (516 lines, 171 Route:: calls — 144 GET, 3 POST, rest via
route()/redirect/view helpers) — the tenant product

subfolders — one component per report type), SaleRequests, Sales

**app/Livewire/Central/** — CPanel (Admins/roles, Blogs, Contacts, Countries,

**app/Livewire/Operations.php** — a loose top-level component (not in a

## 9. SERVICES / REPOSITORIES
================================================================================

### Pattern
Nearly every tenant business entity has: a **Repository** (data access, extends
`BaseRepository`) + a **Service** (business logic, constructor-injects the
Repository, exposes `list/find/save/delete` plus domain-specific methods) +
sometimes a Livewire component consuming the Service directly.

**app/Repositories/BaseRepository.php** implements `App\Interface\BaseInterface`:
`setInstance(Model|Builder $model): void`, `getInstance()`,
`list($relations=[], $filter=[], $perPage=null, $orderByDesc=null)`,
`find(string|int|null $id, $relations=[], $filter=[])`,
`first($relations=[], $filter=[])`,
`findOrFail(string|int $id, $relations=[], $filter=[])`,
`count($filter=[])`, `create(array $data)`, `update(string|int $id, array $data)`,
`delete(string|int $id)`.

Concrete repositories (all `extends BaseRepository`, constructor-inject their
Eloquent model): AccountRepository, AdminRepository, BranchRepository,
BrandRepository, CashRegisterRepository, CategoryRepository,
DiscountRepository, ExpenseCategoryRepository, ExpenseRepository,
FixedAssetRepository, PaymentMethodRepository, ProductRepository,
PurchaseRepository, PurchaseRequestRepository, SaleRequestRepository,
SellRepository, StockRepository, StockTakingRepository,
StockTransferRepository, TaxRepository, TransactionRepository, UnitRepository,
UserRepository — plus **Hrm/** subfolder: AttendanceSheetRepository,
DepartmentRepository, DesignationRepository, EmployeeContractRepository,
EmployeeRepository, ExpenseClaimCategoryRepository, ExpenseClaimRepository,
LeaveRequestRepository, LeaveTypeRepository, PayrollRunRepository,
PayrollSlipRepository. Most just override the constructor; business logic
lives in the Service layer.

Services (28 top-level + 10 Hrm/) generally expose:
`list/activeList($relations=[],$filter=[],$perPage=null,$orderByDesc=null)`,
`find($id=null,$relations=[])`, `count($filter=[])`, `save($id=null,$data)`,
`delete($id)`, plus domain methods, e.g.:
- **AccountService**: `getSupplierAccounts($supplierId)`,
  `getBranchPaymentAccounts(?int $branchId): Collection`,
  `getPaymentAccountsForBranchIds(array $branchIds): Collection`,
  `createAccountForUser($user)`.
- **CashRegisterService**: `getOpenedCashRegister($relations=[])`,
  `increment($id,$field,$amount=1)`, `decrement($id,$field,$amount=1)`.
- **DiscountService**: `saveHistory(Discount $discount, $sale)`.
- **CheckService** (cheque lifecycle, uses DB transactions):
  `collect(int $checkId, ?int $collectedAccountId=null, ?string $note=null): Check`,
  `bounce(int $checkId, ?string $note=null): Check`,
  `clearIssued(int $checkId, ?int $clearedAccountId=null, ?string $note=null): Check`
  — each wraps the state change + related ledger transaction in `DB::transaction()`.

Other services worth noting: `PlanPricingService`, `PlanFeaturePresentationService`
(plan/pricing display logic for central site), `SellService`/`ProductService`/
`StockService`/`StockTakingService`/`StockTransferService`/`PurchaseService`/
`PurchaseRequestService`/`SaleRequestService`/`ExpenseService`/`TaxService`/
`TransactionService`/`UserService`/`UnitService`/`BranchService`/`BrandService`/
`CategoryService`/`PaymentMethodService`/`ExpenseCategoryService`/`FixedAssetService`/
`AdminService`, and Hrm/: `AttendanceSheetService`, `DepartmentService`,
`DesignationService`, `EmployeeContractService`, `EmployeeService`,
`ExpenseClaimCategoryService`, `ExpenseClaimService`, `LeaveRequestService`,
`LeaveTypeService`, `PayrollRunService`, `PayrollSlipService`.

## 10. PAYMENTS
================================================================================

`app/Payments/Interfaces/PaymentMethodInterface`: `pay($data)`,
`capture($transactionId)`, `refund($transactionId)`, `getGrandTotalAmount($amount)`.

Providers implementing it: **Paymob** (Egyptian payment gateway — `pay`,
`capture`, `refund`, plus `createIntention($secret,$integrationIds,$amount,$token)`
and `retriveIntention($publicKey,$clientSecret)`) and **Paypal** (`pay`,
`capture`, `refund`, plus `createAccessToken($clientId,$secret)` and
`createPayment($returnUrl,$cancelUrl,$amount,$currency)`).

`PaymentService` is a thin wrapper: constructor-injects a
`PaymentMethodInterface $gateway` (presumably bound at runtime to Paymob or
Paypal depending on the selected central `PaymentMethod`), exposes
`pay($data)`/`capture($transactionId)`. Used by `PaymentController::callback`
for the subscription checkout flow (central site pricing/checkout ->
Subscription + PaymentTransaction records).

### Middleware (16 total)
Central/general: `AdminTranslationMiddleware`, `CentralWebsitesetLocal`,
`CpanelTranslationMiddleware`, `RedirectToSecure`, `SiteTranslationMiddleware`

whole app — all other validation presumably happens inline in Livewire
components (via Livewire's own `$this->validate()`), which is consistent with
the Livewire-centric architecture but means validation rules are scattered
across 186 files rather than centralized.

### Traits (app/Traits)
- **LivewireOperations** (trait, used across most Livewire components):

  `removeFile($id)`, `redirectWithTimeout($url,$timeout=500)`,
  `redirectToDownload($url)`, `resetFilters()` — a shared UI/UX helper layer
  wrapping livewire-alert + validation + file handling for all CRUD components.
- **Translator** (trait): magic `__get($name)`, `translations()`, `t($key)` —
  in-model/component translation accessor helper.

### Enums (23 files, all backed `enum ... : string`, mostly with `label()` and

### Notifications
`GeneralNotification extends Notification` — generic, constructor-injects
`$translationKey, $translationParams=[]`, implements `via()`/`toArray()` — a

### Mail (8 Mailables, all `implements ShouldQueue`, all with

AdminRegisterRequestMail, ContactUsMail, CustomerDueAmountEmail,

### Imports (7, all `implements ToCollection, WithHeadingRow, WithValidation`,

### Exports
`GeneralExport implements FromCollection` — generic constructor

array-combine header row + mapped data — reused across export features rather

in Livewire — export formatting is centralized while report *querying* is not).

### Console Commands (10 + 1 in Tenant/ subfolder)
`BackfillChecksSetup` (`handle(): int` — one-off data migration for checks
feature), `ClearTenantCache` (iterates `Tenant::all()->each`), `ConvertCurrencies`

`TenantCreateAdmin` (large command — `defaultBranch()`, `defaultPaymentMethods()`,

seeding default branch/payment methods/roles/settings/expense categories).

### Providers
- **AppServiceProvider**: `register()`, `boot()` — sets up `View::share('defaultLayout', ...)`
  (the mechanism selecting which of the 3 UI skins to render) and custom Blade
  directives `@adminCan`/`@endadminCan` (permission-gated Blade sections, likely
  wrapping spatie/laravel-permission checks).
- **HelperServiceProvider**: `register()`, `boot()` — likely just to ensure
  `app/helpers.php` global functions are available / registers helper bindings.
- **TenancyServiceProvider**: `events()` (tenant created/deleted event listeners

  aware update route twice — for two different Livewire asset paths per skin?,
  `View::composer('*', ...)`), `bootEvents()`, `mapRoutes()`,
  `makeTenancyMiddlewareHighestPriority()`.

### View Components
`TableComponent` and `Table5Component` (both `extends Component`,
constructor `(public $columns, public $headers, public $rows=[],

data-table Blade components used across the many list views; the "5" variant
likely a newer/different-column-count iteration — worth checking if
`TableComponent` (non-5) is legacy/should be deprecated in favor of Table5Component.

## 12. DATABASE — MIGRATIONS (filenames only, see section 4 for grouped summary)
================================================================================
Full filename lists are in section 4 (Directory Sitemap → database/). Highlights
for planning purposes:
- Central DB has a cluster of **~15 "seed_*_blogs" migrations all dated
  2026-07-08** (e.g. `seed_cloud_accounting_blogs`, `seed_new_mohaaseb_blogs`,
  `seed_sales_inventory_mohaaseb_blogs`, `seed_erp_pos_difference_blogs`,
  `seed_smarterpguide_comparison_blogs`, `seed_arabic_cloud_accounting_mohaaseb_blogs`,
  etc.) — using migrations to seed marketing blog content is unconventional
  (normally seeders would be used); this inflates the migrations table and
  makes rollback risky. Worth flagging as a candidate to move to a seeder.
- Two duplicate-looking migrations exist: `2026_12_06_153822_add_soft_deleted_to_every_table.php`
  and `2027_12_06_153822_add_soft_deleted_to_every_table.php` (tenant/) — likely
  a copy-paste artifact with a future-dated duplicate; worth checking if both
  are needed/were both actually run.
- A migration filename without a date prefix exists:
  `database/migrations/tenant/add_soft_delete_to_branches_table.php` — breaks
  the normal Laravel timestamp-ordering convention, could cause migration
  ordering issues.
- Tenant DB schema evolved heavily post-launch: many `add_X_to_Y_table`
  migrations retrofitting columns (branch_id, tax fields, deferred revenue,
  soft deletes) onto already-shipped tables — normal iterative development but
  worth knowing the schema has accreted rather than been designed upfront.

## 13. VIEWS STRUCTURE
================================================================================
See section 4 for the full directory breakdown. Key takeaway: `resources/views/livewire/`
(403 files) is triplicated across `admin/`, `hud/`, and `tenant-tailwind-gemini/`
subtrees with **identical folder structure** (accounts, branches, brands,
cash-register, categories, checks, contracting/*, depreciation-expenses,

payables, payment-methods, plans, products, purchase-requests, purchases,
refunds, reports/*, sale-requests, sales, settings, stocks, stock-taking,
taxes, transactions, units, users). This is the single biggest structural fact
for planning improvements: **any UI/UX change to a tenant-app screen likely
needs to be applied in up to 3 places** unless there's a shared partial
strategy already in place. Recommend investigating `AppServiceProvider`'s
`defaultLayout` mechanism and whether the "hud" or "tenant-tailwind-gemini"
skins are legacy/deprecated candidates for removal, or whether one is a WIP
theme migration that should replace the other two once complete.

## 15. SUGGESTED AREAS TO PROBE FURTHER WHEN PLANNING IMPROVEMENTS
================================================================================
1. Confirm whether all 3 view "skins" (admin/hud/tenant-tailwind-gemini) are
   actively used in production or if 1-2 are dead weight — big consolidation
   opportunity if so.
2. Review the `download-file` closure route in tenant.php for path-traversal
   risk (arbitrary `?file=` query param passed to `file_exists`/`response()->download`).
3. Review/remove the `test/{tenant}` debug route in tenant.php — unauthenticated
   tenancy-initializing debug code left in a route file.
4. Investigate why blog content is seeded via ~15 dated migrations instead of
   a seeder — consider migrating that content to `BlogSeeder`.
5. Only one FormRequest class in the entire app — confirm Livewire's own
   `$this->validate()` is consistently used elsewhere and there isn't
   inconsistent/missing validation in older components.
6. Two tenancy config files (`tenancy.php` / `tenancy_live.php`) — confirm which
   is authoritative and whether the other is stale.
7. `TableComponent` vs `Table5Component` — check if the former is legacy.
8. Two near-duplicate "add soft delete to every table" migrations dated a year
   apart — verify no double-application issue.
